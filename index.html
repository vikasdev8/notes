Explain all types of joins in PySpark.UPDATED
★★★★★ Join concepts
+

PySpark supports several join types. Here is a simple guide:

1. Inner Join (how="inner"): Default. Keeps only matching rows from both sides.

df1.join(df2, on="id", how="inner")

2. Left Join (how="left"): Keeps all left rows, plus matching right rows. Unmatched right data becomes null.

df1.join(df2, on="id", how="left")

3. Right Join (how="right"): Keeps all right rows, plus matching left rows.

df1.join(df2, on="id", how="right")

4. Full Outer Join (how="outer"): Keeps all rows from both sides. Also called full or fullouter.

df1.join(df2, on="id", how="outer")

5. Left Semi Join (how="leftsemi"): Keeps left rows that match the right. No right columns are added.

df1.join(df2, on="id", how="leftsemi")

6. Left Anti Join (how="leftanti"): Keeps left rows that do not match the right.

df1.join(df2, on="id", how="leftanti")

7. Cross Join: Matches every left row with every right row. This creates a very large dataset.

df1.crossJoin(df2)
What are the different join strategies in PySpark? Explain with examples.NEW
★★★★★ Execution internals
+

Join strategy determines how Spark physically executes a join. Spark's Catalyst optimizer picks the best strategy based on table sizes and hints. There are 5 strategies:

1. Broadcast Hash Join (BHJ) — Best for joining a small dataset with a large one. The smaller dataset is broadcasted to every executor node, where it is hashed and then joined with the locally held partition of the large dataset. No shuffle of the large side is needed — the join happens entirely locally on each executor.

from pyspark.sql.functions import broadcast
# Force broadcast: small table sent to all executors, hashed, joined locally
large.join(broadcast(small), "key").explain()
# Physical plan shows: BroadcastHashJoin

2. Shuffle Hash Join (SHJ) — Both datasets are shuffled across the cluster so that the same keys from both sides land in the same partitions. Then the smaller side is hashed and bucketed, and joined with the larger side within those partitions. Used when one shuffled side fits in executor memory but is too large to broadcast.

# Force shuffle hash join via hint
df1.hint("shuffle_hash").join(df2, "key").explain()
# Physical plan shows: ShuffledHashJoin

3. Shuffle Sort Merge Join (SMJ) — The default strategy for large–large joins. Involves three phases:
  ① Shuffle — redistribute data by join keys so matching keys go to the same partition.
  ② Sort — order the data within each partition by join key.
  ③ Merge — iterate through the sorted data to join matching keys efficiently.
Reliable but expensive due to the full shuffle and sort cost.

# Default for large-large joins; force with hint
df1.hint("merge").join(df2, "key").explain()
# Physical plan shows: SortMergeJoin

4. Broadcast Nested Loop Join (BNLJ) — Used for non-equi joins (e.g., >, <, <=, range conditions). One entire dataset is broadcasted to all partitions, and a nested loop compares every record of the broadcasted data against every record in the local partition. Very slow — O(n×m) per partition. Spark usually warns when this is chosen.

# Non-equi join — Spark falls back to BNLJ automatically
result = df1.join(df2, df1["start"] <= df2["end"]).explain()
# Physical plan shows: BroadcastNestedLoopJoin

5. Cartesian Product Join — Also used for non-equi joins or when there is no join condition. Instead of broadcasting, the entire partition of one dataset is sent to all other worker nodes to perform a full cross-join against every record of the other dataset. Produces n×m rows — only use intentionally.

# Explicit cross join (requires config to allow)
spark.conf.set("spark.sql.crossJoin.enabled", "true")
df1.crossJoin(df2).explain()
# Physical plan shows: CartesianProduct

Summary table:

Strategy              | Join Type  | Shuffle? | When Used
----------------------|------------|----------|---------------------------
Broadcast Hash Join   | Equi       | No       | One side small (<10 MB)
Shuffle Hash Join     | Equi       | Yes      | One side fits in executor RAM
Shuffle Sort Merge    | Equi       | Yes      | Both sides large (default)
Broadcast Nested Loop | Non-equi   | No       | >/< conditions, slow
Cartesian Product     | Non-equi   | No       | No condition / cross join

Force a strategy with hints: df1.hint("broadcast"), df1.hint("merge"), df1.hint("shuffle_hash"). Always verify with .explain("formatted").

What is the difference between RDD, DataFrame, and Dataset in PySpark?
★★★★★ Asked in almost every interview
+

RDD (Resilient Distributed Dataset) is the low-level API — immutable, distributed collection of objects. No schema. Manual optimization needed.

DataFrame is a distributed table with named columns and schema. Uses Catalyst optimizer for query planning. Most commonly used in DE roles.

Dataset is a typed version of DataFrame (available in Scala/Java; in Python, DataFrame == Dataset[Row]).

For 4 YOE: "I always prefer DataFrames for ETL pipelines due to Catalyst optimization and schema enforcement. RDDs only when I need low-level custom transformations."

Explain lazy evaluation in PySpark and why it matters.
★★★★★ Core concept question
+

Transformations like filter(), select(), groupBy() are NOT executed immediately — they build a DAG (Directed Acyclic Graph). Execution only happens when an action is called (count(), collect(), show(), write()).

Why it matters: Spark's Catalyst optimizer can reorder, merge, and push down operations across the entire plan before executing — often dramatically reducing data scanned and shuffled.

# Transformation (lazy - nothing runs yet)
df2 = df.filter(df.age > 30).select("name", "age")
# Action (triggers DAG execution)
df2.show()
What is the difference between narrow and wide transformations?
★★★★☆ Optimization-focused
+

Narrow transformations: Each input partition maps to exactly one output partition. No data shuffle. Fast. Examples: map(), filter(), select().

Wide transformations: Data moves across partitions (shuffle). Expensive. Examples: groupBy(), join(), distinct(), orderBy().

Key insight: Wide transformations create stage boundaries in the DAG. Minimizing them is a primary optimization strategy.

What is data skew and how do you handle it in PySpark?
★★★★★ Senior-level must-know
+

Data skew happens when some partitions have significantly more data than others, causing certain tasks to take much longer (stragglers).

Strategies:
1. Salting: Add a random prefix to skewed keys to spread load.
2. Broadcast joins: If one table is small, broadcast it to avoid shuffle entirely.
3. Repartition: Force redistribution with df.repartition(n, "key_col").
4. Skew hint (Spark 3+): df.hint("skew", "join_key")

# Salting example
from pyspark.sql.functions import concat, lit, rand
df_salted = df.withColumn("salted_key",
  concat(col("key"), lit("_"), (rand()*10).cast("int")))
What is the difference between cache() and persist()?
★★★★☆ Performance question
+

cache() stores the DataFrame in memory only (shortcut for persist(MEMORY_AND_DISK)).

persist() gives you control over the storage level:
• MEMORY_ONLY
• MEMORY_AND_DISK (default cache)
• DISK_ONLY
• MEMORY_ONLY_SER (serialized, saves memory)

Use when a DataFrame is reused multiple times across actions. Always unpersist() when done.

What is a Broadcast Join and when do you use it?
★★★★★ ETL optimization staple
+

A broadcast join sends the smaller DataFrame to all executor nodes so the join happens locally without a shuffle. Ideal when one side is <10 MB (configurable via spark.sql.autoBroadcastJoinThreshold).

from pyspark.sql.functions import broadcast
result = large_df.join(broadcast(small_df), "key_col")

Spark can also auto-broadcast based on the config threshold. Always set it explicitly for predictability in production.

Explain groupBy() vs Window functions. When do you use window functions?
★★★★★ Coding interview staple
+

groupBy() collapses rows into groups — you lose the original row detail.

Window functions compute aggregates while keeping all original rows — essential for ranking, running totals, lag/lead comparisons.

from pyspark.sql.window import Window
from pyspark.sql.functions import rank, lag

w = Window.partitionBy("dept").orderBy(col("salary").desc())
df.withColumn("rank", rank().over(w)) \
  .withColumn("prev_salary", lag("salary", 1).over(w)).show()
How do you handle duplicate records in a PySpark DataFrame?
★★★★☆ ETL/data quality
+

Three approaches depending on need:

# Drop exact duplicates across all columns
df.dropDuplicates()

# Drop duplicates based on subset of columns
df.dropDuplicates(subset=["id", "event_date"])

# Keep latest record (deduplication with business logic)
from pyspark.sql.window import Window
from pyspark.sql.functions import row_number
w = Window.partitionBy("id").orderBy(col("updated_at").desc())
df.withColumn("rn", row_number().over(w)).filter(col("rn") == 1).drop("rn")
What is repartition() vs coalesce()?
★★★★☆ Performance tuning
+

repartition(n): Full shuffle, can increase or decrease partitions. Used when you need to redistribute data evenly (e.g., before a join or write).

coalesce(n): Merges partitions without full shuffle. Only decreases partition count. More efficient for reducing before a write.

Rule of thumb: Use coalesce before writing to avoid many small files. Use repartition to parallelize better before heavy computations.

What is the difference between partitioning and bucketing in PySpark?NEW
★★★★☆ Storage & join optimization
+

Partitioning creates a folder hierarchy based on column values (year=2024/month=01/). It enables partition pruning — Spark skips irrelevant folders entirely. Use only on low‑cardinality columns (year, region, status).

df.write.partitionBy("year", "month").parquet("/sales/")
# Creates: /sales/year=2024/month=01/part-000.parquet

Bucketing hashes a column into a fixed number of buckets and saves files accordingly. It eliminates shuffles in joins when both sides are bucketed identically. Bucketing requires Hive‑style table (saveAsTable) and works well for high‑cardinality columns like user_id.

df.write.bucketBy(8, "customer_id").sortBy("customer_id") \
  .saveAsTable("sales_bucketed")

Key difference: Partitioning is for filtering (pruning), bucketing is for joining efficiently. Partitioning on high‑cardinality columns creates millions of tiny files — a classic mistake.

How do you read and write Parquet files, and why is Parquet preferred?
★★★★★ Production DE standard
+

Parquet is columnar, compressed, and supports predicate pushdown — ideal for analytical workloads.

# Read
df = spark.read.parquet("s3://bucket/data/")
df = spark.read.option("mergeSchema", "true").parquet("path/")

# Write with partitioning
df.write.mode("overwrite") \
  .partitionBy("year", "month") \
  .parquet("s3://bucket/output/")

Advantages over CSV/JSON: 10–100x faster reads, 75% smaller files, schema embedded in file.

What is the difference between map() and mapPartitions()?NEW
★★★☆☆ RDD-level question
+

map(): Applies a function row-by-row. One function call per record.

mapPartitions(): Applies a function once per partition, receiving an iterator. More efficient when setup/teardown cost is high (e.g., DB connections, ML model loading) — you initialize once per partition, not per row.

def process_partition(iterator):
    conn = open_db_connection()  # opened once per partition
    for row in iterator:
        yield transform(row, conn)
    conn.close()

rdd.mapPartitions(process_partition)

How do you optimize a slow PySpark job?
★★★★★ Scenario / senior question
+

Systematic approach:
1. Check Spark UI: Find stages with long task durations (skew indicator).
2. Fix shuffles: Reduce wide transformations, use broadcast joins for small tables.
3. Fix data skew: Salting, repartitioning.
4. Tune partitions: spark.sql.shuffle.partitions (default 200, often too high for small data, too low for large).
5. Use caching: Cache reused DataFrames.
6. Use columnar formats: Parquet with proper partitioning.
7. Push predicates early: Filter before joins.
8. Avoid UDFs: Native Spark functions are Catalyst-optimized; Python UDFs break optimization.

What is Adaptive Query Execution (AQE) in Spark 3.x?NEW
★★★★★ Modern Spark optimization
+

AQE (Spark 3.0+) re‑optimizes the query plan at runtime based on actual data statistics, not just estimates. It can dynamically change join strategies, coalesce shuffle partitions, and handle skew — all automatically.

Three key features:

1. Dynamic Partition Coalescing: Reduces the number of post‑shuffle partitions when the data is small, avoiding the overhead of many tiny tasks.

2. Automatic Skew Join Handling: Detects skewed partitions and splits them into smaller sub‑partitions, balancing the load. Enabled with spark.sql.adaptive.skewJoin.enabled=true.

3. Dynamic Join Strategy Switching: Can switch a sort‑merge join to a broadcast join if a table turns out to be smaller than expected at runtime.

# Enable AQE (on by default in recent Spark versions)
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")

AQE works best with wide transformations and was a major improvement in Spark 3.0 — it eliminates much of the manual tuning previously needed for partition counts and skew.

What is the Catalyst optimizer and the Tungsten engine?
★★★☆☆ Depth question
+

Catalyst: Spark's query optimizer — the "brain". Applies rule-based and cost-based optimizations: predicate pushdown, column pruning, join reordering. Works on the logical/physical plan before execution.

Tungsten: The "muscle" / execution engine. Manages memory off-heap, generates JVM bytecode on the fly (code generation), and uses cache-aware data structures for better CPU efficiency. Does 3 things: memory management, code generation, caching.

Together they make DataFrames much faster than RDDs.

How do you implement SCD Type 2 in PySpark?
★★★★☆ Real-world ETL scenario
+

SCD Type 2 maintains full history by adding new rows with version flags instead of updating in place.

from pyspark.sql.functions import current_timestamp, lit

# Step 1: Identify changed records
changed = incoming.join(existing, "id") \
  .filter(incoming.value != existing.value)

# Step 2: Expire old records
existing = existing.join(changed.select("id"), "id", "left_anti") \
  .union(existing.join(changed.select("id"), "id") \
    .withColumn("end_date", current_timestamp()) \
    .withColumn("is_current", lit(False)))

# Step 3: Insert new versions
new_rows = changed.withColumn("start_date", current_timestamp()) \
  .withColumn("end_date", lit(None)) \
  .withColumn("is_current", lit(True))
What is spark.sql.shuffle.partitions and how do you tune it?
★★★★☆ Config tuning
+

Controls the number of partitions after a shuffle operation (joins, groupBy). Default is 200.

Rules:
• Too high → many small tasks, overhead
• Too low → large tasks, OOM risk

General formula: Total data size (MB) / 128 is a rough starting point.

spark.conf.set("spark.sql.shuffle.partitions", "400")
# Or in session builder
SparkSession.builder.config("spark.sql.shuffle.partitions", "400")
What is Delta Lake and how does it improve on plain Parquet?
★★★★★ Modern DE stack
+

Delta Lake adds ACID transactions, schema evolution, time travel, and DML (UPDATE/DELETE/MERGE) on top of Parquet.

Key features:
• MERGE (upsert): Essential for SCD and CDC pipelines
• Time travel: Query historical snapshots with VERSION AS OF
• Schema enforcement: Rejects bad data at write time
• Compaction: OPTIMIZE command merges small files

from delta.tables import DeltaTable
dt = DeltaTable.forPath(spark, "s3://bucket/delta_table/")
dt.alias("t").merge(
  incoming.alias("s"), "t.id = s.id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()
Explain PySpark UDFs — why avoid them and what are the alternatives?
★★★★☆ Performance anti-pattern
+

Python UDFs serialize data row-by-row between JVM and Python interpreter — very slow (10–100x vs native functions).

Alternatives (in order of preference):
1. Built-in Spark functions (pyspark.sql.functions) — Catalyst-optimized
2. Pandas UDFs (vectorized UDFs) — operate on Arrow batches, much faster
3. Spark SQL expressions for simple logic

from pyspark.sql.functions import pandas_udf
import pandas as pd

@pandas_udf("double")
def fast_udf(s: pd.Series) -> pd.Series:
    return s * 2.5

df.withColumn("result", fast_udf(col("value")))

What happens during a Spark job — explain stages, tasks, and executors.
★★★★☆ Architecture depth
+

Driver: Orchestrates the job, builds the DAG, schedules tasks.
Executor: Worker process on cluster nodes. Executes tasks and stores cached data.
Job: Triggered by an action. Divided into stages at shuffle boundaries.
Stage: Set of tasks that can run in parallel without shuffle.
Task: Smallest unit of work — processes one partition on one executor core.

Analogy: Job = entire project, Stage = sprint, Task = individual story.

What is predicate pushdown and column pruning in Spark?NEW
★★★★☆ Optimization internals
+

Predicate pushdown: Catalyst moves filter conditions as close to the data source as possible — so fewer rows are read from disk. Works best with Parquet/ORC which support file-level statistics.

Column pruning: Only the columns needed for the query are read from storage. With columnar formats like Parquet, this avoids reading irrelevant columns entirely.

Together they can reduce I/O by 80–90% on large datasets.

# Spark automatically pushes the filter to Parquet read
df = spark.read.parquet("data/")
df.filter(col("year") == 2024).select("name", "salary").show()
# Only reads year=2024 partitions and name+salary columns
How do you handle schema evolution in PySpark pipelines?UPDATED
★★★★☆ Production scenario
+

Schema evolution means your data changes (like adding a new column), and you want Spark to handle it safely.

1. Parquet (mergeSchema)
Combine old and new columns across multiple files.

df = spark.read \
  .option("mergeSchema", "true") \
  .parquet("path/to/files/")

2. Delta Lake
Delta is strict. You must explicitly allow changes.

# Append: Allow adding new columns
df.write.format("delta") \
  .option("mergeSchema", "true") \
  .mode("append").save("path/")

# Overwrite: Replace old schema completely
df.write.format("delta") \
  .option("overwriteSchema", "true") \
  .mode("overwrite").save("path/")

3. Explicit Schema (StructType)
Define exactly what columns exist before reading. Set nullable=True for new columns.

from pyspark.sql.types import StructType, StructField, StringType, IntegerType

my_schema = StructType([
  StructField("id", IntegerType(), True),
  StructField("new_col", StringType(), True)
])

df = spark.read.schema(my_schema).json("path/to/data.json")
What is the difference between sort() and orderBy() in PySpark?NEW
★★★☆☆ Common confusion point
+

sort() and orderBy() are actually aliases — they do the same thing: global sort across all partitions (causes a shuffle).

For performance-sensitive work, use sortWithinPartitions() — it sorts within each partition only, no shuffle, much cheaper. Useful before writing to ensure each output file is internally sorted.

# Global sort (shuffle - expensive)
df.orderBy(col("salary").desc())

# Sort within partitions only (no shuffle - cheap)
df.sortWithinPartitions("dept", "salary")
What is the small files problem in Spark and how do you fix it?NEW
★★★★☆ Production data layout
+

When a Spark job writes many small files (e.g., thousands of files <10 MB each) — common with over‑partitioning or streaming append workloads — it causes excessive metadata overhead, slow listing, and poor read performance.

Fixes:

1. Coalesce/repartition before write – reduce the number of output partitions.

df.coalesce(10).write.parquet("/output/")
# or for even distribution: df.repartition(10).write.parquet("/output/")

2. Delta Lake OPTIMIZE – compaction command that merges small files into larger ones (target ≈1 GB).

from delta.tables import DeltaTable
DeltaTable.forPath(spark, "/delta/table").optimize().executeCompaction()

3. Adjust shuffle partitions – spark.sql.shuffle.partitions can be lowered if the default 200 produces too many small files.

4. Use maxRecordsPerFile – control the number of records per output file.

df.write.option("maxRecordsPerFile", 5000000).parquet("/output/")
What is repartitionByRange() and how is it different from repartition()?NEW
★★★☆☆ Partitioning internals
+

repartition(n, col) uses a hash function on the column to assign rows to partitions. Keys with the same hash land in the same partition, but values are not sorted within or across partitions.

repartitionByRange(n, col) uses range partitioning — it samples the data, determines boundaries, and assigns rows so partition 0 has the lowest range and partition n-1 has the highest. Rows within each partition are in sorted order.

# Hash repartition — rows with same key, same partition but unsorted
df.repartition(8, "customer_id")

# Range repartition — partitions hold sorted, non-overlapping ranges
df.repartitionByRange(8, "order_date")

# Useful before rangeJoin or when writing sorted output per file
df.repartitionByRange(4, "event_date").write.parquet("/output/")

When to use: Use repartitionByRange when downstream operations benefit from sorted data (range joins, sorted writes) or when you want to avoid hot partitions caused by key hash collisions.

What is the difference between checkpoint() and cache() / persist()?NEW
★★★★☆ Lineage & fault tolerance
+

Both store computed data to avoid recomputation, but they differ fundamentally in how lineage is handled.

cache() / persist(): Stores the DataFrame in memory/disk but keeps the full lineage (DAG). If a partition is lost, Spark can recompute it using the lineage. Stored in executor memory — lost when the application ends.

df.cache()     # stores in memory, lineage intact
df.persist(StorageLevel.DISK_ONLY)  # stores on disk, lineage intact

checkpoint()`: Writes the DataFrame to a reliable external storage (HDFS/S3) and truncates the lineage. Spark no longer tracks how the data was computed. Persists beyond the Spark session.

spark.sparkContext.setCheckpointDir("s3://bucket/checkpoints/")
df.checkpoint()   # lineage is cut; data saved to HDFS/S3

When to use checkpoint: For very long iterative pipelines (ML training loops, graph algorithms) where the lineage DAG becomes so long it causes memory issues on the driver. Also used in Structured Streaming for state recovery.

Key difference summary: Cache preserves lineage (recomputable); checkpoint breaks lineage (smaller DAG, external durability).

What are Accumulators in PySpark and when do you use them?NEW
★★★☆☆ Shared variable / debugging
+

Accumulators are shared variables that executors can only add to (write) while the driver can read the final aggregated value. They are used to count events, errors, or metrics during distributed processing without collecting all data to the driver.

from pyspark.sql import SparkSession
spark = SparkSession.builder.getOrCreate()
sc = spark.sparkContext

# Create a Long accumulator
error_count = sc.accumulator(0)
null_count  = sc.accumulator(0)

def validate_row(row):
    global error_count, null_count
    if row["age"] is None:
        null_count += 1
    if row["salary"] < 0:
        error_count += 1
    return row

# Use in an RDD action (not a transformation — accumulators in
# transformations can double-count on re-execution)
df.rdd.foreach(validate_row)

print("Nulls found:", null_count.value)
print("Errors found:", error_count.value)

Important gotcha: Accumulators inside transformations (like map) can be counted multiple times if Spark re-executes tasks. Always trigger them inside actions (like foreach) for accurate counts.

What are Broadcast Variables in PySpark? How are they different from broadcast joins?NEW
★★★★☆ Shared state optimization
+

Broadcast variables let you ship a read-only value (like a dictionary or lookup table) to all executors once, rather than sending it with every task. They are used in RDD-level operations. Broadcast joins (using broadcast(df)) are the DataFrame-level equivalent but are handled by Spark automatically.

sc = spark.sparkContext

# Create a broadcast variable with a lookup dict
country_map = {"IN": "India", "US": "United States", "UK": "United Kingdom"}
bc_map = sc.broadcast(country_map)

# Use in RDD map — bc_map.value is available on every executor
rdd = sc.parallelize([("IN", 100), ("US", 200), ("UK", 150)])
result = rdd.map(lambda x: (bc_map.value.get(x[0], "Unknown"), x[1]))
result.collect()
# [('India', 100), ('United States', 200), ('United Kingdom', 150)]

# Always unpersist when done to free memory
bc_map.unpersist()

DataFrame broadcast join (for comparison):

from pyspark.sql.functions import broadcast
result = large_df.join(broadcast(small_df), "key")

Key difference: sc.broadcast() is for RDD operations with arbitrary Python objects. broadcast(df) is for DataFrame joins and is handled by Catalyst. Both avoid re-sending data per task.

How do you handle OutOfMemory (OOM) errors in PySpark?NEW
★★★★★ Production firefighting
+

OOM in Spark typically happens in two places: executor OOM (during task execution) or driver OOM (when collecting data). Each has different causes and fixes.

Executor OOM — common causes & fixes:

1. Data skew — one partition is huge. Fix: salt the key or use AQE skew handling.

2. Too few partitions — each partition has too much data. Fix: increase shuffle partitions.

spark.conf.set("spark.sql.shuffle.partitions", "800")

3. Broadcast of a too-large table — disable auto-broadcast or lower the threshold.

spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "-1")  # disable

4. Caching too much — evict cached DataFrames. Use MEMORY_AND_DISK so spill to disk when memory is full.

df.persist(StorageLevel.MEMORY_AND_DISK)

Driver OOM — common causes & fixes:

Calling collect(), toPandas(), or creating huge broadcast variables. Fix: never collect large datasets to driver; use show(), limit(), or write to storage instead.

# BAD — pulls all data to driver
data = df.collect()

# GOOD — write to storage or sample
df.write.parquet("/output/")
df.limit(100).toPandas()  # only for small previews

General tuning levers: Increase executor memory (spark.executor.memory), increase memory fraction (spark.memory.fraction), enable off-heap (spark.memory.offHeap.enabled).

What is Dynamic Partition Pruning (DPP) in Spark 3.x?NEW
★★★★☆ Modern optimizer feature
+

Dynamic Partition Pruning (DPP) is a Spark 3.0+ optimization where Spark pushes a filter from the result of a subquery into a partitioned table scan — at runtime — eliminating irrelevant partitions that cannot be determined at planning time.

Classic example: You join a large fact table (partitioned by date_id) with a small dimension table filtered by country = 'IN'. Without DPP, Spark scans all partitions of the fact table. With DPP, it first evaluates which date_id values the filtered dimension returns, then scans only those partitions.

# fact table is partitioned by date_id
fact   = spark.read.parquet("/warehouse/fact_sales/")
dim    = spark.read.parquet("/warehouse/dim_date/")

# DPP kicks in automatically here (Spark 3+)
result = fact.join(dim.filter(dim.country == "IN"), "date_id")
result.explain()
# Look for "PartitionFilters" applied dynamically in the physical plan

Requirements for DPP to work:

1. The large table must be partitioned on the join key.
2. The small side must be broadcasted (or small enough for Spark to broadcast).
3. spark.sql.optimizer.dynamicPartitionPruning.enabled = true (default in Spark 3).

spark.conf.set("spark.sql.optimizer.dynamicPartitionPruning.enabled", "true")

DPP can eliminate 90%+ of data scanned in typical star-schema queries.

How do you use explain() in PySpark to read query plans?NEW
★★★★☆ Debugging / optimization
+

df.explain(mode) prints the query execution plan. Reading it correctly helps diagnose slow jobs. Spark 3+ added named modes.

Available modes:

"simple" (default) — Physical plan only. Good for quick checks.

# == Physical Plan ==
# *(2) HashAggregate(...)

"extended" — Shows Parsed → Analyzed → Optimized Logical Plan + Physical Plan. Best for understanding what Catalyst did.

What is the difference between client mode and cluster mode in Spark?NEW
★★★★☆ Deployment / production
+

The deploy mode determines where the Driver process runs.

Client mode: The Driver runs on the machine that submitted the job (e.g., your laptop or a gateway node). Logs and output are visible locally. Suitable for interactive development and debugging.

spark-submit --master yarn --deploy-mode client my_job.py

Cluster mode: The Driver runs inside the cluster (on one of the worker/manager nodes). The submitting machine can disconnect after submission. Suitable for production scheduled jobs.

spark-submit --master yarn --deploy-mode cluster my_job.py

Key differences:

• Network latency: In client mode, the driver communicates with executors over the network — bad if the submitting machine is far from the cluster.
• Fault tolerance: In cluster mode, losing the submitting machine doesn't kill the job.
• Logs: In cluster mode, logs go to the cluster's log system (YARN logs, Spark History Server).
• Databricks/EMR: Managed services abstract this — you always get cluster mode effectively.

Rule of thumb: Use client mode for development/notebooks; use cluster mode for production pipelines.

What is Z-Ordering in Delta Lake and how does it improve query performance?NEW
★★★★☆ Delta Lake optimization
+

Z-Ordering (ZORDER) is a multi-dimensional data clustering technique in Delta Lake. It co-locates related data in the same set of files so that Spark can skip more files during a query (data skipping).

How it works: Z-Order maps multiple column values to a single Z-order curve value and sorts files by that value. Files that are close on the Z-curve contain similar values across all Z-ordered columns — enabling effective data skipping even for multi-column filters.

from delta.tables import DeltaTable

# Run OPTIMIZE with ZORDER on frequently filtered columns
DeltaTable.forPath(spark, "/delta/sales") \
  .optimize() \
  .executeZOrderBy("customer_id", "order_date")

# Subsequent queries filter-push more aggressively:
spark.read.format("delta").load("/delta/sales") \
  .filter((col("customer_id") == 42) & (col("order_date") > "2024-01-01")) \
  .show()
# Spark skips files that don't contain customer_id=42 or that date range

Z-Ordering vs Partitioning:

• Partitioning: Creates physical directory splits. Works well for low-cardinality columns (year, region). Querying by partition = folder pruning.
• Z-Ordering: Works within existing partitions for high-cardinality columns (user_id, product_id). Does not create new directories. Relies on Delta's file-level statistics for data skipping.
• Combine both: Partition by year/month, Z-Order by customer_id for best results.

When to use: Z-Order columns you filter on frequently but cannot partition on (high cardinality). Re-run OPTIMIZE ... ZORDER BY periodically as new data arrives (VACUUM old files after).

🔥 Most Asked Core Operations
What is the difference between loc and iloc?
★★★★★ Asked in almost every Pandas round
+

loc: Label-based indexing. Uses actual index labels or column names. Slice is inclusive on both ends.

iloc: Integer position-based (0-indexed). Slice is exclusive on the right (Python convention).

df.loc['a':'d', 'age']       # rows with labels 'a' through 'd' (inclusive)
df.loc[df['age'] > 25]       # boolean mask
df.iloc[0:3, 2]              # rows 0,1,2 — column at position 2 (NOT col 3)
df.iloc[-1]                  # last row

Key gotcha: With a default integer index, df.loc[0:3] returns 4 rows (0,1,2,3) while df.iloc[0:3] returns 3 rows (0,1,2).

How do you filter rows using multiple conditions? Explain query() vs boolean indexing.NEW
★★★★★ Daily Pandas usage
+

Boolean indexing – combine conditions with & (and), | (or), ~ (not). Always wrap each condition in parentheses.

query() – string expression, cleaner for complex filters. Supports @variable injection and uses numexpr for speed.

min_age = 25
# Boolean indexing
df[(df['age'] > 25) & (df['dept'] == 'Engineering')]

# query() — equivalent, cleaner
df.query("age > @min_age and dept == 'Engineering'")

# NOT condition
df[~df['status'].isin(['inactive', 'deleted'])]
df.query("status not in ['inactive', 'deleted']")

When to prefer query(): Complex multi-condition filters where chained &/| becomes unreadable.

What is SettingWithCopyWarning and how do you fix it?
★★★★☆ Common production bug
+

Occurs when you assign to a slice that Pandas suspects may be a copy of the original DataFrame. The assignment silently does nothing on the original data — a subtle bug.

# ❌ Problematic — may not update original df
df[df['age'] > 25]['salary'] = 0

# ✅ Use .loc — always modifies the original
df.loc[df['age'] > 25, 'salary'] = 0

# ✅ Explicit copy when you intentionally want a separate DataFrame
subset = df[df['age'] > 25].copy()
subset['salary'] = 0

In Pandas 2.0+ enable pd.options.mode.copy_on_write = True for safer defaults that eliminate this warning class entirely.

How do you handle missing values in Pandas?
★★★★★ Data quality staple
+

Detection: df.isna().sum() — count per column. df.info() — shows non-null counts.

Removal: df.dropna(subset=['critical_col']) — drops rows where specified column is null.

Filling strategies:

# Fill with median
df['age'].fillna(df['age'].median())

# Column-specific fill
df.fillna({'status': 'unknown', 'amount': 0})

# Forward fill (time series — carry last known value forward)
df['col'].fillna(method='ffill')

# Linear interpolation
df['col'].interpolate(method='linear')

Best practice: Always explain the business reason — dropping loses data; filling introduces assumptions.

How do you remove duplicate records in Pandas?NEW
★★★★☆ Data quality / ETL
+
# Exact duplicates across all columns
df.drop_duplicates()

# Duplicates based on a subset of columns
df.drop_duplicates(subset=['user_id', 'event'])

# Keep latest record (sort first, then dedup on key)
df.sort_values('updated_at', ascending=False) \
  .drop_duplicates('user_id', keep='first')

# Mark duplicates without removing (keep=False marks ALL occurrences)
df['is_dup'] = df.duplicated(subset=['id'], keep=False)
Explain groupby() with multiple aggregations using agg().
★★★★★ Coding question
+
# Single aggregation
df.groupby('dept')['salary'].mean()

# Multiple aggregations on a single column
df.groupby('dept')['salary'].agg(['mean', 'max', 'min'])

What is transform() vs agg() in groupby()?NEW
★★★★★ Coding interview staple
+

agg() – reduces each group to a single summary row (smaller DataFrame).
transform() – returns a Series with the same index as the original, broadcasting group stats back to every row.

# agg — collapses to one row per group
df.groupby('dept')['salary'].agg(['mean', 'max'])

# transform — keeps all original rows, adds dept avg as new column
df['dept_avg_salary'] = df.groupby('dept')['salary'].transform('mean')

# Now compute pct of dept salary — all rows preserved
df['pct_of_dept'] = df['salary'] / df['dept_avg_salary']

Rule: Add group stats to original rows → transform(). Need a summary table → agg().

How do you filter entire groups with groupby().filter()?NEW
★★★☆☆ GroupBy advanced
+

filter() keeps only the groups where a condition returns True. The entire group is kept or dropped — not individual rows.

# Only departments with more than 5 employees
df.groupby('dept').filter(lambda g: len(g) > 5)

# Only departments where average salary > 50000
df.groupby('dept').filter(lambda g: g['salary'].mean() > 50000)

# Groups where at least one employee is senior
df.groupby('dept').filter(lambda g: (g['level'] == 'senior').any())

Important: Returns original rows (not aggregated). Result is a subset of the full DataFrame.

What is the difference between apply(), map(), and applymap()?
★★★★☆ Functional operations
+

Series.map() – element-wise on a single Series. Best for value substitution or lookup dictionaries.
DataFrame.apply() – applies a function along rows (axis=1) or columns (axis=0). Receives a Series.
DataFrame.map() (formerly applymap in Pandas <2.1) – element-wise on every cell of the DataFrame.

# map: replace values using a dict
df['grade'] = df['score'].map({90: 'A', 80: 'B', 70: 'C'})

# apply: row-wise custom computation
df['total'] = df.apply(lambda x: x['price'] * x['qty'], axis=1)

# DataFrame.map: round every cell
df[['a','b']].map(lambda x: round(x, 2))

Performance note: All three are slower than vectorized operations. Prefer np.where(), np.select(), or built-in Series methods when possible.

How do you use np.select() for multiple conditions (vectorized if-else)?NEW
★★★★☆ Vectorization best practice
+

np.select() is a vectorized way to assign values based on multiple conditions — much faster than chaining np.where() or using .apply().

import numpy as np

conditions = [df['amount'] > 1000, df['amount'] > 500]
choices    = ['high',              'mid']
df['label'] = np.select(conditions, choices, default='low')

# Equivalent slow approach with apply (avoid for large DataFrames)
df['label'] = df['amount'].apply(
    lambda x: 'high' if x > 1000 else 'mid' if x > 500 else 'low'
)

Key: Conditions are evaluated in order — first match wins. Always provide a default for rows matching none.

Explain merge(), join(), and concat() with examples.
★★★★★ Very common coding question
+

merge() – SQL-style join on columns or indexes. Most flexible.
join() – joins on index by default. Convenient shortcut when DataFrames share an index.
concat() – stacks DataFrames vertically (append rows) or horizontally without key matching.

# merge on a common column — left join
pd.merge(orders, customers, on='customer_id', how='left')

# join on index
df1.join(df2.set_index('id'), on='id')

# concat vertically (append rows)

# concat horizontally (side by side)

How do you validate merges to catch unexpected data loss?NEW
★★★★☆ Production data quality
+

Use indicator=True in merge() — it adds a _merge column showing whether each row came from left_only, right_only, or both.

result = pd.merge(left, right, on='id', how='outer', indicator=True)
result['_merge'].value_counts()
# left_only     12   ← rows only in left (unmatched)
# right_only     5   ← rows only in right (unmatched)
# both         983   ← matched rows

# Assert no data loss
assert (result['_merge'] == 'both').all(), "Unmatched rows found!"

Also use the validate parameter to enforce join cardinality:

pd.merge(df1, df2, on='id', validate='one_to_one')
# Raises MergeError if duplicates exist on key in either side
What is pivot_table() and how does it differ from pivot()?NEW
★★★★☆ Reshaping / reporting
+

pivot() – reshapes without aggregation. Fails if there are duplicate entries for the same index/column combination.
pivot_table() – handles duplicates with an aggregation function (default mean). Supports fill_value for missing combinations.

# pivot — fails on duplicate (region, product) pairs
df.pivot(index='region', columns='product', values='revenue')

# pivot_table — safe with duplicates; aggregates and fills gaps
df.pivot_table(
    index='region', columns='product', values='revenue',
    aggfunc='sum', fill_value=0
)

Explain pd.cut() vs pd.qcut().NEW
★★★★☆ Feature engineering
+

pd.cut() – bins data into equal-width intervals that you define (based on value ranges).
pd.qcut() – bins data into equal-frequency quantile bins (each bin gets roughly the same number of records).

# pd.cut — fixed value boundaries
df['age_group'] = pd.cut(
    df['age'], bins=[0, 18, 35, 60, 100],
    labels=['teen', 'young', 'mid', 'senior']
)

# pd.qcut — equal-frequency quantiles
df['salary_quartile'] = pd.qcut(
    df['salary'], q=4, labels=['Q1','Q2','Q3','Q4']
)

# Check distribution
df['salary_quartile'].value_counts()  # each bucket ≈ equal count

Use cut for domain-driven ranges (age groups). Use qcut for percentile-based segmentation (top/bottom quartile).

Explain rolling(), expanding(), and ewm().NEW
★★★★☆ Moving averages & smoothing
+

rolling(n) – fixed-size sliding window of last n rows.
expanding() – cumulative window growing from the first row (all history).
ewm(span=n) – exponentially weighted; recent values have a higher weight.

df['ema_14']  = df['sales'].ewm(span=14).mean()     # EMA over 14 periods

# Rolling with min_periods (avoid NaN when window not full)
df['3d_sum'] = df['sales'].rolling(3, min_periods=1).sum()

When would you migrate from Pandas to PySpark or Dask?NEW
★★★★★ Senior-level design decision
+

Pandas – ideal for data that fits in single-machine memory (< a few GB) and for quick exploration or ad-hoc analysis.

PySpark – the choice when data exceeds 10 GB, requires distributed storage (S3, HDFS), or needs production fault tolerance and parallelism across a cluster.

Dask – provides a Pandas-like API that scales across cores or a small cluster. Good for intermediate datasets (< ~100 GB on one machine) without the full Spark overhead.

# Pandas — works fine in memory
df = pd.read_parquet('data.parquet')   # 2 GB file — ok

# Dask — parallel Pandas across multiple cores
import dask.dataframe as dd
df = dd.read_parquet('data/*.parquet') # 50 GB — distributed lazily
df.groupby('dept')['salary'].mean().compute()

# PySpark — true distributed cluster
spark.read.parquet('s3://bucket/data/').groupBy('dept').mean('salary')

Rule of thumb: If a single pd.read_csv() causes MemoryError or takes more than 5 minutes, move to PySpark or Dask. If you need ACID transactions or streaming, use PySpark + Delta Lake.
